Adopting the new newsletter style in the codebase

The mockup in newsletter-preview.jsx is one component. Your codebase renders the newsletter differently — Jinja templates per provider, plus a shared theme CSS. To adopt the new style without rewriting your pipeline, treat it as a new theme + updated provider templates. No service code changes are required.

How your codebase actually renders the newsletter

From app/services/care_circle/:

  • session_assembler.py orchestrates the daily session, calling each enabled provider in order.
  • provider_base.BaseCareCircleProvider renders each provider's templates/default.html through Jinja, then prepends get_combined_theme_css(theme_name) as an inline <style> block.
  • Theme CSS lives in app/services/care_circle/providers/themes/ — currently classic.css, high_contrast.css, soft_pastel.css, plus master_online.css / master_print.css.
  • Per-provider templates live in each provider folder, e.g. providers/this_day_history/templates/default.html. They emit <table class="section section--<key>"> with semantic classes (section__title, section__body, etc).
  • Header/footer are themselves providers: newsletter_header and newsletter_footer.
  • Email delivery: newsletter_email_service.py takes the assembled HTML, runs Premailer to inline styles, and sends.
Implication: the design tokens that drive the look — colours, type scale, masthead treatment, card backgrounds, provenance pills — should be expressed as a new theme CSS file that overrides class hooks already in the per-provider templates. You only need to touch templates where the markup itself needs to change (masthead, family blocks, provenance badges).

The plan, in five steps

Step 1

Add a new theme: ellis.css (or ink_quill.css)

Create app/services/care_circle/providers/themes/ellis.css. Translate the tokens from newsletter-preview.jsx's NL_CSS into class-based rules that map onto the existing template hooks (.section, .section__title, .section__body, .section--<key>).

Use container queries on the outer wrapper so the same CSS gives 1-column on phone / narrow email and 2-column on desktop / wide email — exactly as the mockup does.

/* themes/ellis.css — sketch */
.cc-newsletter { container-type: inline-size; background: #fcfaf6; color: #231913;
  font-family: Georgia, "Times New Roman", serif; }
.section { border-radius: 14px; padding: 16px 18px; background: #f4ebd9;
  border: 1px solid rgba(35,25,19,0.06); margin-bottom: 14px; }
.section__title { font-family: Georgia, serif; font-weight: 600;
  font-size: clamp(18px, 2cqw, 22px); line-height: 1.15; margin: 8px 0 0; }
.section__body  { font-size: clamp(14px, 1.5cqw, 16px); line-height: 1.55; }

/* Theme variants per provider key */
.section--newsletter_header { /* broadsheet masthead — see Step 3 */ }
.section--this_day_history  { border-left: 4px solid #d86c3d; background: rgba(31,59,54,0.04); }
.section--garden_almanac    { background: rgba(182,198,155,0.22); }
.section--family_voice      { background: linear-gradient(135deg,#f1a57b,#d86c3d); color: #fff; }

/* Two-column body on wide containers */
@container (min-width: 720px) {
  .cc-cols { display: grid; grid-template-columns: 1.4fr 1fr; gap: 32px; }
}

Step 2

Wire the theme into config

The selector is already there — BaseCareCircleProvider reads theme from the patient config and looks for themes/<name>.css. Set Rose's patient config (or the global default) to "theme": "ellis". No Python changes needed.

If you want it in the admin theme dropdown, add "ellis" to the theme_map in provider_base.py around line ~205 (the dict that maps stems to display names).

Step 3

Update the masthead in providers/newsletter_header/templates/default.html

The current header is a navy-blue panel. Replace it with the broadsheet markup from the mockup — keep the same Jinja variables (patient_name, family_name, generated_at) so the provider's Python doesn't change.

<!-- Broadsheet masthead, email-safe table version -->
<table class="section section--newsletter_header cc-mast" width="100%" cellpadding="0" cellspacing="0" border="0">
  <tr><td align="center" style="border-bottom:3px double #231913; padding-bottom:12px;">
    <div class="cc-mast__eyebrow">{{ generated_at }}</div>
    <div class="cc-mast__title">The {{ family_name or patient_name }} Family Daily</div>
    <div class="cc-mast__foot">For {{ patient_name }}</div>
  </td></tr>
</table>

Style .cc-mast__title etc. in ellis.css.

Step 4

Add a provenance helper and emit it from each provider template

Two options — pick one:

4a. Jinja macro (recommended). Add app/services/care_circle/templates/_macros/provenance.html with a {% macro prov(kind, who) %}…{% endmacro %} that emits the inline-SVG badge from the mockup. Import it at the top of each provider template:

{% from "_macros/provenance.html" import prov %}
<table class="section section--this_day_history">
  <tr><td>
    {{ prov('curated', 'On this day · ' ~ year) }}
    <h2 class="section__title">{{ event }}</h2>
  </td></tr>
</table>

The Jinja env in provider_base.py uses FileSystemLoader(self.template_dir) per-provider, so to share the macro you'll need to add a second loader path. Easiest: change the render_template environment to FileSystemLoader([self.template_dir, SHARED_TEMPLATES_DIR]) (one-line edit, ~line 340 in provider_base.py).

4b. Pure-CSS pill. If you'd rather not touch Python, give every provider template a data-prov="auto|curated|family" attribute on its outer table and render the badge with a ::before pseudo-element using a base64 SVG background-image. Simpler but pseudo-elements get stripped by some email clients — Gmail in particular.

Convention for which kind to use:

kindused by
autoweather, garden_almanac, joke, recipe, this_day_history, puzzle, song_of_the_day
curatedhymn_of_the_day, daily_blessing, seasonal_poem, world_news
familyfamily_greeting, letter_to_family, memory_lane_photo, family voice notes

Step 5

Wrap the assembled HTML in the container-query root

In session_assembler.py, the assembled body is currently a stack of <table> sections. Wrap them in a single <div class="cc-newsletter">…</div> so container-type: inline-size has somewhere to live.

For the two-column body layout, group the "main" providers and the "sidebar" providers in <div class="cc-cols"><div class="cc-main">…</div><div class="cc-side">…</div></div>. Decide which provider keys go where via a small list in the assembler — or add a "column": "main" | "side" field to each provider's config.json and group on that.

Email caveat: Outlook on Windows uses Word's renderer and ignores @container queries. For Outlook fall back gracefully — the single-column stack is what they'll see, which is fine. Premailer (already in your pipeline) will inline the rest of the styles. Do not put critical structure inside container queries.

How to validate

  • Use scripts/build_care_circle_sample_newsletter.py to generate runtime/artifacts/care-circle-sample-newsletter-rose-ellis.html and open it in a browser at desktop and phone widths — should reflow exactly like the mockup.
  • Use scripts/send_test_newsletter_patient5.py to send through Premailer + your real email path; check Gmail (web + mobile) and Outlook web. Outlook desktop is allowed to fall back to single-column.
  • Run scripts/validate_care_circle_newsletter_uniqueness.py to make sure the theme didn't break section selectors used in tests.
